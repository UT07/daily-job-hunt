# Deep Second-Pass Audit — 2026-04-29

> **Scope:** read-only audit covering 10 dimensions (auth, GDPR, WebSocket apply, `_db None` sites, daily-batch request models, resume version history, SFN `$.foo` references, Lambda IAM, CORS/cookies, EventBridge cron). Findings exclude items already in flight via Areas A–D of the comprehensive-prod-health plan and the three audits dated 2026-04-28.

> **Severity:** **P0** = security/data-loss/prod-broken; **P1** = visibly-broken-or-confusing for users; **P2** = code-smell, hardening opportunity. All file:line references are against `origin/main` at `fc50be3`.

## Top P0 findings (one-liner each)

1. **`JobHuntApi` Lambda has no `ecs:RunTask`/`ecs:StopTask` IAM** — Apply Start/Stop Session calls boto3 ECS but the SAM role only has SQS+S3+SFN. Apply path is a ticking AccessDenied bomb. `template.yaml:1399-1409`.
2. **`WsDisconnectFunction` calls `ecs.stop_task` without `ecs:StopTask` IAM** — Fargate browser tasks are not cleaned up on WS disconnect; leaks $$ until idle-timeout reaper. `template.yaml:1611-1613` vs `lambdas/browser/ws_disconnect.py:39`.
3. **`resume_versions` table has no UNIQUE(user_id, job_id, version_number)** — every restore/re-tailor reinserts a row with the same `version_number`; user sees multiple "v1" rows because the FIRST tailored output never inserts a row, then both `restore` and `re-tailor` snapshot the live state under `current_version` (still 1) before bumping. `supabase/migrations/008_resume_versions.sql:1-13`.
4. **WebSocket token TTL is 60 s but Fargate cold-start is typically 30–90 s** — both frontend and browser tokens are minted before `ecs.run_task`; by the time Fargate is ready and the browser worker dials the WS, the `browser`-aud token may have already expired. `shared/ws_auth.py:17` + `app.py:2778-2779`.
5. **Daily EventBridge cron sends `{"user_id": "default"}`** — there is no `users` row with id `default`. `LoadUserConfig` reads from `user_search_configs` keyed by user_id; the daily job effectively runs against an empty config. `template.yaml:1310`.

## Findings by dimension

### 1. Auth flow end-to-end

- **F-AUTH-1 / P1** — `LoginPage.jsx:125` and `ResetPasswordPage.jsx:119,128` use `minLength={6}`; `Settings.jsx:297` rejects `< 8`. Cluster B branch `f7e1b33` already addresses this — keep on Area D's cherry-pick path. Already in flight.
- **F-AUTH-2 / P2** — `LoginPage.jsx:25` calls `signUp(email, password)` then `setSuccess('Account created. Check your email...')`. Supabase's signUp returns success even if email confirmation is *not* required (depends on Supabase Auth settings). If email confirmation is disabled, the message is misleading; if enabled, no client-side check verifies the user actually went through confirmation before showing the LoginPage and proceeding. `web/src/pages/LoginPage.jsx:24-26`.
- **F-AUTH-3 / P1** — `LoginPage.jsx:24-29` has no race protection: rapid double-submit while `loading=true` is gated by the button state, but `handleSubmit` itself doesn't check `loading`. With Enter-key submit + slow network, two `signUp` calls can fire and you'll get a dup-account error masking the real one. `web/src/pages/LoginPage.jsx:16-35`.
- **F-AUTH-4 / P2** — `useAuth.js:39-43` calls `supabase.auth.updateUser({ password })` but never invalidates other sessions. Any other device logged in keeps its access token valid for the full Supabase token lifetime. Recommend calling `signOut({ scope: 'others' })` after a password change.
- **F-AUTH-5 / P2** — `AuthProvider.jsx:23` does `getSession().then(...)` without `.catch()`; if Supabase storage is corrupted (rare but happens with Safari ITP), `loading` never flips to `false` and the app gets stuck on the spinner. `web/src/auth/AuthProvider.jsx:23-27`.
- **F-AUTH-6 / P2** — `useAuth.js:26` sets `redirectTo: window.location.href` for Google OAuth — if the user hits Google sign-in from `/login?next=/dashboard`, the redirect comes back to the same URL which is fine, but if they hit it from `/onboarding` the OAuth provider sees the path and redirect callback lands them mid-onboarding even if onboarding wasn't complete. Should redirect to `/login` and let the route guard re-decide.
- **F-AUTH-7 / P1** — `ResetPasswordPage.jsx:25` checks `!user` and shows "Link expired" — but Supabase populates the user only after the recovery hash is processed. If the hash arrives but the `useEffect` in `AuthProvider` hasn't fired yet, this page can flicker the "expired" message before the recovery session lands. Wait for `loading=false` AND `user=null` for at least one tick before declaring expired.

### 2. GDPR endpoints

- **F-GDPR-1 / P1** — `gdpr.py:13` — `record_consent` writes `gdpr_consent_at` but does not verify the user EXISTS in the `users` table first. If the auth user's row was never created (signup race; we have no on-signup trigger that I can see), `update_user` returns no rows, the endpoint returns 200, but consent is silently lost. `gdpr.py:13-15`.
- **F-GDPR-2 / P0** — `gdpr.py:67-71` — "delete my account" is **only** a soft-delete that sets `gdpr_deletion_requested_at`. There is no scheduled hard-delete job referenced in `template.yaml` (no EventBridge rule invokes `hard_delete_user`). The 30-day clock never fires in production. **GDPR Art 17 not actually implemented.**
- **F-GDPR-3 / P1** — `gdpr.py:79-106` — `hard_delete_user` takes `s3_client` arg but the parameter is named ambiguously: line 90 says `bucket = s3_client` but treats it as a string bucket name. Code is dead today (#F-GDPR-2) but if/when it runs, the API contract is wrong — caller passes a boto3 client expecting deletion, gets a string-vs-client TypeError. `gdpr.py:87-91`.
- **F-GDPR-4 / P1** — `gdpr.py:18-64` — export ZIP includes `resumes`, `search_config`, `jobs`, `runs`, `profile` — but **does not include `applications`, `application_timeline`, `resume_versions`, `audit_log`, or `feedback`**. GDPR Art 15 says the user gets ALL data; missing tables are non-compliance.
- **F-GDPR-5 / P2** — `gdpr.py:36` — `default=str` on `json.dumps` is a coarse fallback that silently turns datetimes into strings but also turns binary blobs (e.g., `bytes` if any column ever returns bytes) into Python repr like `b'\\x00...'`. Future schema additions could leak weird strings. Use a typed serializer.
- **F-GDPR-6 / P1** — `app.py:2436-2462` — none of the three GDPR endpoints accept a confirmation flag (e.g., `?confirm=DELETE_FOREVER`). A single misclick on a Settings page button could trigger soft-delete with no client-side guardrail in the API contract.
- **F-GDPR-7 / P2** — `app.py:2438` — `gdpr_export` returns the ZIP body inline; for a power user with thousands of jobs this could exceed Lambda's 6 MB sync response limit. Should write to S3 and return a presigned URL.

### 3. WebSocket apply path + eligibility gate

- **F-WS-1 / P0** — `app.py:2748` — `apply_start_session` calls `ecs.run_task(...)` and `template.yaml:1399-1409` shows the JobHuntApi role lacks `ecs:RunTask` permission. Same Lambda lacks `ecs:StopTask` for `apply_stop_session`. **The whole apply path is broken in prod the moment a user clicks Apply.** Fix: add an inline policy granting `ecs:RunTask` on the browser TaskDef ARN + `ecs:StopTask` on `arn:aws:ecs:...:task/<browser-cluster>/*` + `iam:PassRole` for the BrowserTaskRole.
- **F-WS-2 / P0** — `lambdas/browser/ws_disconnect.py:39` calls `boto3.client("ecs").stop_task(...)` but `template.yaml:1611-1613` only attaches `DynamoDBCrudPolicy`. **Stop-task fails silently** (handler swallows in line 44-45 with `logger.warning`) — Fargate browsers leak when the WS drops.
- **F-WS-3 / P0** — WS-token TTL race (top-5). `shared/ws_auth.py:17` `_DEFAULT_TTL = 60`. `app.py:2778-2779` issues both tokens before `ecs.run_task` returns. Fargate cold-start regularly exceeds 60s. Both tokens may be expired by the time the WS connects. Fix: bump default TTL to 300 s or mint browser token lazily after Fargate starts.
- **F-WS-4 / P1** — `lambdas/browser/ws_route.py:38-39` — when peer is gone, returns 200 with body "no peer". The frontend has no signal that its message was dropped. Add an in-band reply telling the frontend the browser-side dropped.
- **F-WS-5 / P1** — `lambdas/browser/ws_disconnect.py:46-47` — `update_status(session_id, "ended")` runs even if the OTHER side is still connected and just lost a redundant tab. Should only mark "ended" when `not other_conn`. The current code IS guarded by line 34 (`if not other_conn`), but the inner block flows through the unconditional logger.info. OK on re-read — clear, ignore.
- **F-WS-6 / P1** — `app.py:2760-2774` — "reuse existing session" path returns the pre-existing token ONLY for the frontend role. There is no way to mint a fresh frontend token if the original expired (60 s TTL). User reloads the page after 70 s → reuse hits but token is dead → WS rejects. Fix: reissue the frontend token on every start-session call.
- **F-WS-7 / P1** — `app.py:2470-2521` — `apply_eligibility` does NOT check `gdpr_deletion_requested_at`. A user who pressed "Delete account" then comes back can still queue applications.
- **F-WS-8 / P2** — `lambdas/browser/ws_connect.py:51` — checks `session.get("user_id") != claims["sub"]` — good. But there is no rate limit; a user can spam `$connect` to scan session IDs. Cheap mitigation: increment a counter in DynamoDB and reject after N attempts/minute.
- **F-WS-9 / P1** — `app.py:2737-2830` — there is no row-lock when checking `find_active_session_for_user`. Two parallel start-session calls can both pass the check and both run a Fargate task. Cost: $0.20–$0.40 per duplicate task before the WS-disconnect cleanup catches up.

### 5. `run_pipeline` request models

- **F-PIPE-1 / P1** — `app.py:1904-1940` — `run_pipeline` request only carries `queries` from `PipelineRunRequest`. But the SFN's `LoadUserConfig` overwrites `$` with the entire user config, so `queries` from request body is **discarded**. Either remove the field from the request model (it's unused) or make `LoadUserConfig` merge instead of replace. Currently the field is actively misleading.
- **F-PIPE-2 / P0** — `template.yaml:1310` `Input: '{"user_id": "default"}'` for the daily EventBridge target. There is no real user with ID `default` — `LoadUserConfig` will load defaults (queries=["software engineer"], min_score=60) per `lambdas/pipeline/load_config.py:70-73`. So the daily run scrapes for a synthetic user, scores them, but **`SaveJob` writes against `user_id="default"`** which is foreign-key-orphaned. Worse, the scraped jobs land in `seen_jobs` under `user_id="default"` and become invisible to real users. **Almost certain root cause for "no daily auto-run hits any real user's dashboard."** Fix: either run one execution per active user, or rename to a synthetic per-user fan-out.
- **F-PIPE-3 / P1** — `app.py:1943-1998` — `run_single_job` does not propagate `apply_url` (Cluster B's stray `c406976` addresses receive but not the SFN-input thread-through). Already in Area C scope.

### 6. Resume version history

- **F-VER-1 / P0** — `supabase/migrations/008_resume_versions.sql:1-13` — no UNIQUE(user_id, job_id, version_number). User-reported "multiple v1 entries for same job" is the inevitable consequence. Fix: `ALTER TABLE resume_versions ADD CONSTRAINT uq_resume_versions UNIQUE (user_id, job_id, version_number);` plus dedupe migration.
- **F-VER-2 / P0** — `app.py:1683-1700` (`restore_resume_version`) and `app.py:2222-2239` (`re_tailor_job`) use IDENTICAL pattern: read `current_version`, snapshot live URLs into `resume_versions` with `version_number=current_version`. **First-ever tailor run does NOT insert v1**, so the very first restore/re-tailor inserts v1 retroactively. If the user has clicked re-tailor multiple times before any restore, you'll have v1, v2, v3 each as a snapshot — but if the live `resume_version` was bumped without the snapshot insert (e.g., one of the 2222-2234 try/except swallows fired), the table is left with duplicate `version_number`s. Fix: assign `version_number = (max existing + 1)` atomically using a SELECT…FOR UPDATE, or use `INSERT ... ON CONFLICT (job_id, version_number) DO NOTHING` once the unique constraint is in.
- **F-VER-3 / P1** — `app.py:2233-2234` — `try/except` swallows insertion failure with a warning, then unconditionally bumps `resume_version` and starts a new SFN execution. If the insert failed, the user has a "version bump" with no historical row. Should re-raise or roll back the version bump.
- **F-VER-4 / P1** — `app.py:1685-1692` — snapshots current resume URLs, but the `resume_s3_url` is a **presigned URL** that expires in days. If the user restores months later, the version row points to a 403. Should store `resume_s3_key` (the immutable key) and re-presign on read.
- **F-VER-5 / P1** — `app.py:1645-1707` — `restore_resume_version` does not include the **score data** (`avg_score`, `tier`) in the snapshot or restore. After restoring a v1 resume, the live `jobs.avg_score` still reflects the v3 tailoring. Either restore the score too, or add a per-version score column.
- **F-VER-6 / P2** — There's no UI/API to **delete** a version. The table grows without bound. Add cleanup or cap at 10/job.

### 8. Lambda IAM roles

- **F-IAM-1 / P0** — `JobHuntApi` (template.yaml:1394-1409) lacks `ecs:RunTask`, `ecs:StopTask`, `iam:PassRole` for the browser cluster. Fix: add inline policy mirroring StepFunctionsRole's RunFargateTasks block (lines 692-714) but pointing at `BrowserTaskDef`.

- **F-IAM-4 / P1** — `JobHuntApi` has `S3CrudPolicy(BucketName=utkarsh-job-hunt)` — `S3CrudPolicy` is bucket-wide CRUD. For follow-up: scope to per-prefix policies (e.g., separate `users/`, `confirmations/`, `sessions/`) — over-broad today.
- **F-IAM-5 / P2** — `WsConnectFunction.Environment.SUPABASE_JWT_SECRET` is plumbed in (line 1594) but the secret should ideally come from SecretsManager not as a CFN parameter (currently `SupabaseJwtSecret` is `NoEcho` string-param — fine but rotates require stack update).
- **F-IAM-6 / P2** — Most scrapers (template.yaml:90-244) only have `SSMParameterReadPolicy("naukribaba/*")`. They need to write nothing. Confirmed least-privilege. ✓
- **F-IAM-7 / P1** — `BrowserTaskRole` (template.yaml:1668-1697) — gives Fargate `dynamodb:GetItem`, `dynamodb:UpdateItem` but NOT `dynamodb:PutItem`. If `shared/browser_sessions.py` ever has to create a session row from inside Fargate (e.g., session row was wiped while task was running), the put would fail. - **F-IAM-8 / P1** — `SendEmailFunction` (template.yaml:401-413) only has `SSMParameterReadPolicy`. Email sending in `lambdas/pipeline/send_email.py` uses Gmail SMTP via app password (which IS in SSM). OK. ✓
- **F-IAM-9 / P0** — No IAM role explicitly grants the `JobHuntApi` Lambda the ability to invoke `BrowserSessionsTable` (DynamoDB), AND there's no env var pointing it at the table name (search `SESSIONS_TABLE` in template.yaml — it's only set on the WS Lambdas, not JobHuntApi). ## Additional cross-cutting findings

- **F-X-1 / P1** — `audit_middleware.py` is wired in `app.py:134` BEFORE `set_audit_db(_db)` is called in `lifespan` startup. First request after cold-start may hit middleware while `_db is None`.
